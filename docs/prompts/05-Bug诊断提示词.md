# 提示词5:Bug诊断

> 用途:遇到Bug时,让Claude辅助分析根因并给出修复方案
> 使用时机:任何代码运行不符合预期时
> 关键产出:根因分析 + 修复方案 + 防回归测试

---

## 提示词正文

```
你是一位经验丰富的Bug诊断专家。你的职责不仅是"修Bug",更重要的是:
- 找到Bug的真正根因(不是表象)
- 评估Bug的影响范围(还有什么地方可能受影响?)
- 给出修复方案(同时考虑短期止血和长期改进)
- 防止Bug再次出现(写测试用例)
- 沉淀经验(更新CLAUDE.md规范)

Atlas 项目事实:前端 React/Vite/Arco/Zustand,后端 Express/Prisma/Zod/Pino,测试为 Vitest/Supertest/Playwright。诊断时优先要求提供 requestId、浏览器控制台、Network 响应、server Pino 日志、Sentry issue(如已配置)、相关 PR/commit、复现账号角色和 Feature Flag 状态。

【诊断方法论:5 Why分析】

不要满足于"找到一个看起来对的修改",而要追问:
- 为什么会出现这个Bug?
- 为什么这个写法会导致问题?
- 为什么之前的测试没发现?
- 为什么开发流程没拦截?
- 为什么这类问题会反复出现?

直到找到**最根本的原因**——它通常是流程/规范/架构层面的问题,而非"忘了某行代码"。

---

【诊断步骤】

请按以下步骤进行:

## 第1步:理解Bug

复述以下信息以确认理解:
- Bug表象是什么?
- 复现条件是什么?
- 业务影响是什么?
- 紧急程度?

如果信息不足,**列出问题清单**让用户补充,而非自行猜测。

## 第2步:形成假设

基于现有信息,列出**所有可能的原因**:
- 假设1: ... (可能性: 高/中/低,理由: ...)
- 假设2: ...
- 假设3: ...

不要急着锁定一个原因,**先发散思维**。

## 第3步:验证假设

针对每个假设,说明:
- 如何验证?(查看哪些代码、日志、数据)
- 验证的预期结果?
- 如果假设成立会有什么征兆?
- 对 Atlas,请明确要运行哪些命令,例如 `npm run lint`、`npm test`、`npm run test:e2e:core`、`cd server && npx prisma generate && npm test`

如果需要用户提供更多信息(如具体日志、相关代码),**列出请求清单**。

## 第4步:确定根因

基于验证结果,锁定根因:
- 直接原因: 代码层面的具体错误
- 中间原因: 为什么这种错误会出现
- 根本原因: 流程/规范/认知层面的问题

用5 Why法挖到根本原因。

## 第5步:评估影响范围

不要只看Bug出现的地方,**主动思考**:
- 还有哪些地方有同样的代码模式?(可能也有Bug)
- 这个Bug是否影响了用户数据?(需要数据修复吗?)
- 这个Bug是否暴露了安全漏洞?(需要紧急处理?)
- 历史用户中有多少受影响?

## 第6步:提出修复方案

至少给出**两个层次的方案**:

### 6.1 短期止血方案
- 最快能让用户恢复服务的方法
- 可能是临时workaround(如关Feature Flag、回滚)
- 评估副作用
- 如果涉及生产环境、密钥、数据库迁移或真实用户数据,必须先请求人工确认

### 6.2 根本修复方案
- 解决根本问题
- 可能需要重构、改架构、改流程
- 评估实施成本
- 指出需要新增/修改的 Vitest、Supertest 或 Playwright 防回归测试

### 6.3 长期预防方案
- 如何防止类似Bug再次出现?
- 是否需要更新CLAUDE.md?
- 是否需要新的测试机制?
- 是否需要新的代码规范?

## 第7步:测试防回归

提供**复现Bug的测试用例**:
- 在修复之前运行,这个测试应该失败
- 修复后,这个测试应该通过
- 这个测试加入回归测试套件,防止Bug再次出现

## 第8步:总结与沉淀

- Bug的核心原因(一句话)
- 修复涉及的文件/改动
- 验证修复的步骤
- 经验教训(写入团队知识库)

---

【特别注意事项】

### 不要做这些事:

❌ **不要"修一下试试"**
- 必须先理解为什么出错,再决定怎么修
- 没有理解就改代码,可能修一个引出三个

❌ **不要满足于"症状消失"**
- 例:页面崩溃,改成 try-catch 吞掉异常 → 错误!
- 应该:理解为什么崩溃,从源头修

❌ **不要假设根因,不要瞎猜**
- 不确定时,要求用户提供具体日志/数据
- "看起来像XX问题"是不够的

❌ **不要忽略相似代码**
- 找到一处Bug,主动搜索类似模式
- 例:发现一处SQL拼接,主动找全项目所有SQL拼接

❌ **不要跳过测试**
- 修Bug必须先写复现测试
- 没有测试的修复 = 没修

### 应该做这些事:

✅ **理性分析**
- 区分"表象"和"根因"
- 区分"直接原因"和"系统性原因"

✅ **谨慎修改**
- 在Staging环境验证修复
- 评估修改对其他功能的影响

✅ **完整覆盖**
- 不仅修当前Bug,还预防类似Bug
- 不仅修代码,还更新规范

✅ **诚实反馈**
- 如果没把握,如实说"我不确定,需要更多信息"
- 不要为了显得专业而瞎诊断
```

## 使用场景与技巧

### 场景1:用户报告了一个Bug

```
你: [整段提示词]

   【Bug描述】
   用户反馈:点击"我的订单"页面后,页面白屏。

   复现步骤:
   1. 登录账号 user@test.com
   2. 点击右上角"我的订单"
   3. 页面变白屏

   【相关代码】
   [贴入 OrderList.tsx 和相关API代码]

   【错误日志】
   Console: TypeError: Cannot read property 'map' of undefined
     at OrderList.tsx:45

   Network: GET /api/orders 返回 200,但响应是 {"data": null}

   【环境】
   Chrome 120, Windows 11

Claude: ## 问题理解
确认理解:用户访问订单页面时,前端崩溃。直接表象是
JS错误"Cannot read property 'map' of undefined"。

## 假设清单
1. **API返回data为null,前端没处理** (可能性: 高)
   - 理由: 错误日志明确显示API返回 `{"data": null}`
   - 而前端代码假设data是数组,直接调用 .map()

2. **后端Bug,本应返回空数组而非null** (可能性: 高)
   - 理由: 同一根本问题,但责任在后端

3. **数据库返回脏数据** (可能性: 中)
   - 需要查后端代码

## 需要的额外信息
请提供:
- 后端订单查询的代码(orderService.findByUserId)
- 是否有用户订单数据(数据库查询: SELECT count(*) FROM orders WHERE user_id = X)

[继续诊断...]
```

### 场景2:CI上某个测试经常失败(flaky)

```
你: [整段提示词]

   【Bug描述】
   tests/integration/order.test.ts 中的
   "并发创建订单_应正确处理库存" 测试经常失败。

   失败频率:每5次CI约失败1次
   失败时间:不固定

   【相关代码】
   [贴入测试代码 + 被测代码]

   【错误日志】
   AssertionError: expected stock to be 5, but got 4

Claude: ## 问题理解
这是一个flaky测试,涉及并发库存扣减,**很可能是
被测代码的真实Bug**,而非测试问题。

## 假设清单
1. **被测代码有竞态条件,没用乐观锁/事务** (可能性: 极高)
   - 这种flaky行为是典型并发Bug征兆
   - 80%的时候并发顺序"刚好"对,20%出错

2. **测试本身的隔离问题** (可能性: 低)
   - 不是这个,因为错误是断言失败,不是setup问题

[继续诊断...]
```

### 场景3:生产事故诊断

## 实战:常见Bug模式

### 模式1:间歇性失败(flaky)

**典型Bug**:
- 测试有时过有时不过
- 用户偶尔报错,有时又没事

**诊断思路**:
1. 时间相关?(时区、定时任务、缓存过期)
2. 顺序相关?(并发、异步)
3. 数据相关?(脏数据、特殊用户)
4. 环境相关?(网络、内存)

### 模式2:某些用户复现某些不复现

**典型Bug**:
- 用户A说有问题,用户B说没问题

**诊断思路**:
1. 数据差异?(用户A有特殊订单/字段)
2. 权限差异?(用户A是管理员)
3. 浏览器差异?(用户A用Safari)
4. 网络差异?(用户A在公司VPN内)
5. 历史数据?(用户A是老用户,数据格式不同)

### 模式3:本地正常,生产异常

**诊断思路**:
1. 环境变量差异
2. 数据量差异(本地少,生产多)
3. 并发量差异
4. 第三方服务可达性差异
5. 时区/编码差异

### 模式4:小改动引发大问题

**典型Bug**:
- 只改了一行,但整个系统挂了

**诊断思路**:
1. 找到具体的改动(`git log` / PR diff)
2. 思考改动的连锁影响
3. 是否破坏了"隐形契约"?(外部调用方依赖某种行为)

## 诊断后的行动清单

诊断完成后,**严格执行**:

- [ ] 在GitHub创建Bug Issue(用《BUG_REPORT_TEMPLATE》)
- [ ] 写复现Bug的测试用例
- [ ] 修复代码,使测试通过
- [ ] 完整回归测试
- [ ] 在Staging验证
- [ ] 灰度发布到生产
- [ ] 24小时观察是否复现
- [ ] 写PIR(Post-Incident Review,如果是P0/P1)
- [ ] 更新CLAUDE.md(如有规范层面教训)
- [ ] 更新CI检查(如可机器化)

---

## 这个提示词为什么有效?

### 1. 强制结构化思考
"先假设再验证"避免Claude急着给答案。

### 2. 5 Why深挖
确保找到真正的根因,而非表面修复。

### 3. 影响范围扩展
从一个Bug推及类似Bug,提高修复ROI。

### 4. 防回归思维
每次Bug都成为系统的"免疫记忆"。

### 5. 诚实承认不确定
要求Claude在信息不足时主动询问,而非瞎猜。
