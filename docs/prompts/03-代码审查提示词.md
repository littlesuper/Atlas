# 提示词3:代码审查(每天必用)

> **业务人员日常质量保障的核心工具**
> 用途:让Claude(新对话)严格审查Claude(原对话)生成的代码
> 使用时机:每次让AI写完代码,在提交PR前必须执行
> 核心理念:**双AI互审**——生成代码的Claude会偏袒自己,新对话的Claude没有这个偏见

---

## 为什么这个提示词最重要?

业务人员看不懂代码,**唯一的把关方式就是让另一个AI严格审查**。

这套机制的有效性,完全取决于:
1. **必须用新对话**:绝不在写代码的对话里做review
2. **必须严格执行**:每次都用,不能"看起来没问题就跳过"
3. **必须修复🔴问题**:严重问题不修复就不能合并

---

## 提示词正文(请复制以下全部内容)

```
你是一位以严苛著称的资深软件工程师,同时具备:
- 15年安全工程经验,见过各种生产事故
- 5年Code Review经验,审查过10万+ PR
- 性能优化专家,熟悉数据库、缓存、并发
- 代码规范洁癖,任何"看起来还行但有隐患"的代码都不放过

现在请你审查以下代码。你的态度应该是:
- 找问题,而不是夸奖
- 设想最坏情况(用户输入恶意数据、网络中断、数据库挂了、并发冲突)
- 不放过任何细节(命名、注释、错误处理、日志)
- 即使是Claude生成的代码也不留情面——AI也会犯错

---

【审查清单】请按以下顺序逐项检查,**每一项都要给出明确结论**:

## 🔴 1. 安全性(零容忍,任何问题都是阻断级)

### 1.1 注入攻击
- [ ] SQL拼接?必须使用参数化查询(ORM/Prepared Statement)
- [ ] NoSQL注入?(MongoDB的$where等)
- [ ] 命令注入?(child_process.exec不带参数化)
- [ ] LDAP注入?
- [ ] XPath注入?
- [ ] 模板注入?(SSTI)

### 1.2 跨站攻击
- [ ] XSS:用户输入直接渲染?innerHTML / dangerouslySetInnerHTML?
- [ ] CSRF:状态变更操作有CSRF Token保护?
- [ ] Clickjacking:有X-Frame-Options头?

### 1.3 身份认证
- [ ] 密码是否哈希存储?(bcrypt/argon2,成本因子≥12)
- [ ] Token是否安全生成?(crypto.randomBytes,不是Math.random)
- [ ] Token是否有有效期?是否能刷新?
- [ ] 登出后Token是否失效?
- [ ] 是否有暴力破解防护?(速率限制、账号锁定)

### 1.4 授权
- [ ] 每个敏感操作是否检查权限?
- [ ] 是否存在越权访问?(用户A能访问用户B的数据吗?)
- [ ] 是否存在垂直越权?(普通用户能调用管理员接口吗?)
- [ ] 资源所有权是否校验?

### 1.5 敏感信息
- [ ] 代码中是否硬编码密钥/密码/Token?
- [ ] 日志中是否打印敏感信息?(密码、银行卡、身份证)
- [ ] 错误信息是否泄露内部细节?(堆栈、SQL、文件路径)
- [ ] API响应是否返回多余敏感字段?(密码哈希、内部ID)
- [ ] URL查询参数是否包含敏感数据?

### 1.6 输入验证
- [ ] 所有外部输入是否验证?(请求体、查询参数、Headers、Cookie)
- [ ] 文件上传是否限制类型/大小?
- [ ] 是否有反序列化漏洞?(不可信数据反序列化)

### 1.7 依赖安全
- [ ] 引入的新依赖是否有已知漏洞?
- [ ] 依赖是否还在维护?

## 🔴 2. 正确性(业务核心,严重问题阻断)

### 2.1 边界情况
- [ ] 空输入(null/undefined/空字符串/空数组)
- [ ] 零值(0, 0.0)
- [ ] 负值(-1, 负数边界)
- [ ] 极大值(MAX_INT, 超长字符串)
- [ ] 特殊字符(emoji, Unicode, 控制字符)
- [ ] 边界数字(浮点精度,如0.1+0.2!=0.3)

### 2.2 并发与竞态
- [ ] 是否存在竞态条件?(多请求同时修改同一资源)
- [ ] 是否使用了乐观锁/悲观锁/事务?
- [ ] 是否有重复提交防护?(idempotency)
- [ ] 异步操作的顺序是否正确?

### 2.3 错误处理
- [ ] 所有可能抛错的代码是否被捕获?
- [ ] 错误是否记录日志(含上下文)?
- [ ] 错误是否给用户友好提示?
- [ ] 是否吞掉了错误?(catch后什么都不做)
- [ ] 异步错误是否处理?(unhandledRejection)

### 2.4 事务完整性
- [ ] 多步操作是否在事务中?
- [ ] 事务是否正确回滚?
- [ ] 事务范围是否合理?(不要太大也不要太小)
- [ ] 是否处理了死锁?

### 2.5 数据完整性
- [ ] 是否有数据校验?(类型、范围、格式)
- [ ] 关键字段是否有数据库约束?(NOT NULL, UNIQUE, FK)
- [ ] 是否处理了脏数据?(已存在数据可能不符合新规则)

### 2.6 时间与时区
- [ ] 时间是否使用UTC?
- [ ] 时区是否正确处理?
- [ ] 夏令时是否考虑?
- [ ] 闰年/闰秒是否考虑?

## 🟠 3. 性能(影响生产,需要修复)

### 3.1 数据库
- [ ] 是否存在N+1查询?(循环里查询)
- [ ] 是否使用了索引?(WHERE/ORDER BY字段)
- [ ] 是否有全表扫描?
- [ ] 是否JOIN太多表?(>5个警惕)
- [ ] 是否SELECT *?(应只取需要的字段)
- [ ] 是否分页?(大结果集必须分页)

### 3.2 算法复杂度
- [ ] 时间复杂度是否合理?(避免O(n²)以上,除非n小)
- [ ] 是否在循环中做昂贵操作?(数据库查询、网络请求)
- [ ] 是否重复计算?(应缓存结果)

### 3.3 内存
- [ ] 是否一次性加载大量数据?(应流式或分批)
- [ ] 是否有内存泄漏?(事件监听器未清理、闭包持有大对象)
- [ ] 大对象是否及时释放?

### 3.4 网络
- [ ] HTTP请求是否过多?(应批量或合并)
- [ ] 是否使用了缓存?(HTTP cache, Redis)
- [ ] 静态资源是否优化?(压缩、CDN)

### 3.5 并发
- [ ] 同步操作是否阻塞了请求处理?(应异步)
- [ ] 是否有合理的连接池?
- [ ] 异步任务是否合理调度?

## 🟡 4. 可维护性(代码质量)

### 4.1 命名
- [ ] 变量/函数名是否表意清晰?
- [ ] 是否有缩写/单字母变量?(除循环索引外避免)
- [ ] 是否使用项目约定的命名规范?
- [ ] Boolean变量是否is/has/can开头?

### 4.2 函数与文件
- [ ] 单个函数是否过长?(>50行警惕)
- [ ] 函数是否单一职责?
- [ ] 文件是否过长?(>500行警惕)
- [ ] 类的职责是否清晰?

### 4.3 复杂度
- [ ] 嵌套是否过深?(>3层警惕)
- [ ] 圈复杂度是否合理?(>10警惕)
- [ ] 是否有Magic Number?(应提取为常量)

### 4.4 重复代码
- [ ] 是否有相似代码?(应提取共用函数)
- [ ] 是否有相同的字符串/常量?

### 4.5 注释与文档
- [ ] 公共API是否有JSDoc/Docstring?
- [ ] 复杂逻辑是否有注释?
- [ ] 注释是否解释"为什么"而非"做什么"?
- [ ] 是否有过期/错误的注释?

### 4.6 类型(TypeScript/Python)
- [ ] 是否使用了any/object/unknown?
- [ ] 类型是否准确?
- [ ] 是否避免了类型断言(as)?

### 4.7 项目规范一致性
- [ ] 是否遵守CLAUDE.md中的规范?
- [ ] 文件位置是否合理?
- [ ] 是否使用项目约定的工具/库?

## 🟡 5. 测试覆盖

- [ ] 是否有测试代码?
- [ ] 测试是否覆盖核心逻辑?
- [ ] 测试是否覆盖异常路径?
- [ ] 测试是否独立?(不依赖其他测试)
- [ ] 测试是否快速?(无sleep)
- [ ] 测试是否有意义?(不是为覆盖率而写)

## 🟢 6. 用户体验(前端代码)

- [ ] 是否有loading状态?
- [ ] 是否有错误状态?
- [ ] 是否有空状态?
- [ ] 是否有响应反馈?
- [ ] 是否考虑了无障碍?(aria-label, 键盘操作)
- [ ] 是否考虑了响应式?
```

## 使用流程详解

### 步骤1:开新对话(关键!)

❌ **错误做法**:在生成代码的对话里说"请审查这段代码"
- 因为生成代码的Claude会"为自己辩护",找不到自己的错

✅ **正确做法**:**完全开新对话**,新Claude没有偏见

### 步骤2:准备材料

需要准备4样东西:
1. **本提示词正文**(整段)
2. **需求摘要**(让Claude知道代码该做什么)
3. **完整代码**(所有相关文件,不要只贴一部分)
4. **CLAUDE.md内容**(让Claude按你们的规范审查)

### 步骤3:整体粘贴

把上面"提示词正文"框里的全部内容复制下来,**替换最后4个占位符**:
- `[在这里贴入相关需求摘要]`
- `[在这里贴入完整代码]`
- `[在这里贴入测试代码]`
- `[在这里贴入相关的CLAUDE.md内容]`

### 步骤4:接收报告

Claude会输出严格的审查报告。

### 步骤5:分级处理

根据报告:
- 🔴 阻断级问题:**必须修复**才能合并
- 🟠 严重问题:**强烈建议修复**
- 🟡 改进建议:**酌情处理**(可记录issue后续优化)

### 步骤6:回到原对话修改

把审查报告的🔴和🟠问题,回到**原对话**(写代码的Claude)说:

```
代码审查发现以下问题,请修复:

[贴入🔴和🟠问题清单]

请按建议修改后,给出新版代码。
```

### 步骤7:再审查

修改后,**再用新对话**做一次审查,确认问题已修复且没有引入新问题。

直到没有🔴和🟠问题,才能提交PR。

## 实战示例

### 示例:审查一个登录API

**输入给Claude:**

```
[整段提示词]

【需求背景】
用户登录API,验证邮箱+密码,成功返回JWT Token。
连续3次失败锁定账户15分钟。

【代码】
async function login(req, res) {
  const { email, password } = req.body;
  const user = await db.query(`SELECT * FROM users WHERE email = '${email}'`);
  if (user && user.password === password) {
    const token = Math.random().toString(36);
    res.json({ token, user: user });
  } else {
    res.status(401).json({ error: 'Invalid credentials' });
  }
}

【测试代码】
(无)

【项目规范】
(贴入CLAUDE.md安全相关内容)
```

**Claude会输出:**

```
# Code Review 报告

## 总体评价
🔴 严重问题,绝对不能合并

## 🔴 阻断级问题

### 问题1: SQL注入漏洞
**位置**: login 函数,第3行
**类型**: 安全
**描述**: 直接用字符串拼接SQL,如果email包含 `' OR '1'='1`,
        将绕过密码校验,任何人都能登录任何账号。
**风险**: 数据库被拖、所有用户账号被盗
**建议修改**:
const user = await db.query('SELECT * FROM users WHERE email = $1', [email]);

### 问题2: 密码明文比较
**位置**: 第4行
**类型**: 安全
**描述**: user.password === password 说明数据库存的是明文密码。
        一旦数据库泄露,所有密码全部暴露。
**风险**: 用户密码大规模泄露
**建议修改**:
const isValid = await bcrypt.compare(password, user.passwordHash);

### 问题3: Token生成不安全
**位置**: 第5行
**类型**: 安全
**描述**: Math.random()不是密码学安全的随机数,Token可被预测。
**风险**: Token伪造,攻击者冒充用户
**建议修改**:
const token = jwt.sign({ userId: user.id }, JWT_SECRET, { expiresIn: '1h' });

### 问题4: 响应泄露敏感信息
**位置**: 第6行
**类型**: 安全
**描述**: 整个user对象返回,包含密码哈希、内部字段等。
**风险**: 信息泄露
**建议修改**:
res.json({ token, user: { id: user.id, email: user.email, name: user.name }});

### 问题5: 缺少账号锁定机制
**位置**: 整体逻辑
**类型**: 正确性
**描述**: 需求要求"连续3次失败锁定15分钟",代码完全没实现。
**建议修改**: 需要追踪登录失败次数和锁定状态。

### 问题6: 没有输入验证
**位置**: 第2行
**类型**: 安全 + 正确性
**描述**: email和password没有验证,空值/超长/格式错都没处理。
**建议修改**: 用Zod schema验证。

### 问题7: 没有错误处理
**位置**: 整个函数
**类型**: 正确性
**描述**: 数据库查询失败、网络异常都会让进程崩溃。
**建议修改**: try-catch包裹,记录日志。

### 问题8: 完全没有测试
**位置**: 整个文件
**类型**: 测试
**描述**: 没有任何测试,无法保证修改后还能工作。
**建议修改**: 至少添加正常登录、密码错误、SQL注入防御、锁定机制测试。

## 📝 给业务人员的话

这段代码存在严重安全漏洞,**任何人都能登录任何账号**。
绝对不能合并,必须让Claude重新生成,并明确要求:
1. 使用参数化查询(防SQL注入)
2. 密码用bcrypt哈希(防泄露)
3. Token用jwt + 加密随机(防伪造)
4. 实现账号锁定(防暴力破解)
5. 添加完整测试

请回到生成代码的对话,把以上要求贴给Claude,要求重新生成。
```

看到了吗?**Claude生成的代码可能有8个严重漏洞**,如果业务人员直接上线,后果不堪设想。

这就是为什么这个提示词是**业务人员日常的核心工具**。

## 进阶用法

### 用法1:针对性审查

如果你只关心某个方面,可以聚焦:

```
请重点审查以下代码的【安全性】,其他方面可简略:

[代码]
```

### 用法2:对比审查

如果Claude修改了代码,可以让另一个Claude对比新旧版本:

```
以下是修改前的代码:[旧代码]
以下是修改后的代码:[新代码]

请审查:
1. 修复是否到位?
2. 是否引入了新问题?
3. 是否有未发现的旧问题被忽略?
```

### 用法3:深入特定问题

```
针对你提出的【SQL注入】问题,请详细解释:
1. 攻击者具体如何构造payload?
2. 攻击成功后能做什么?
3. 防御方案的工作原理?
4. 还有哪些类似的攻击需要防御?
```

## 这个提示词为什么有效?

### 1. 角色锚定
让Claude扮演"严苛的资深工程师",激发"找茬"模式。

### 2. 详尽清单
不依赖Claude的"自由发挥",每一项都明确列出来。

### 3. 强制分级
🔴/🟠/🟡 强制Claude区分严重程度,而非"都说有点小问题"。

### 4. 双AI互审
新对话的Claude没有"为自己辩护"的偏见,会更挑剔。

### 5. 明确格式
结构化输出方便业务人员理解和处理。

## 心态建设

### 业务人员要相信:

✅ Claude会犯错,而且经常犯错
✅ 代码越复杂,越可能有隐藏问题
✅ 严格的审查不是浪费时间,是节省时间(后期修bug更贵)
✅ 我看不懂代码没关系,我懂得"必须严格审查"就够了

### 业务人员要警惕:

❌ "Claude生成的代码看起来很专业,应该没问题"——错觉!
❌ "我看了一下,没看出问题,应该OK"——你没看出≠没问题
