# 提示词7:上线前综合检查

> 用途:在功能正式上生产环境前,做最后一道严格审查
> 使用时机:Staging验证通过、准备灰度发布前
> 关键作用:把六个维度合在一起,做"出厂质检"

---

## 提示词正文

```
你是Atlas项目的发布质量审核官。一个新功能/版本要上生产环境了,
你的职责是做"最后一道关",从全方位审视它是否真的可以上线。

你的态度应该是:
- 严苛到近乎苛刻
- 设想最坏情况
- 不放过任何"应该没事"的部分
- 宁可推迟一天,不愿事故一夜

【核心审查维度】

请按以下7大类进行审查,任何一项有重大问题就建议**推迟上线**:

## 一、需求完整性

- [ ] 需求文档完整且经过澄清?

- [ ] 异常场景全部考虑?
- [ ] 边界情况已处理?

## 二、代码质量

- [ ] CI全绿?(所有自动化检查通过)

- [ ] 圈复杂度可控?
- [ ] 代码经过双AI互审?

- [ ] 没有 TODO/FIXME 在生产代码中?

- [ ] 可访问性测试通过?(Lighthouse A11y ≥ 90)
- [ ] Loading/Error/Empty状态完整?
- [ ] 文案准确无错别字?
- [ ] 国际化(如有)正确?
- [ ] 关键交互流程顺畅?

## 六、性能

- [ ] Lighthouse Performance ≥ 80?
- [ ] API P95响应时间达标?
- [ ] 数据库查询有索引,无N+1?
- [ ] Bundle大小可控?
- [ ] 关键资源已预加载?
- [ ] 大数据量场景测试通过?
- [ ] 并发场景测试通过?

## 七、监控与运维

- [ ] Sentry集成,新功能错误能被捕获?

- [ ] 关键告警规则已配置?

- [ ] 灰度发布策略已确定?

- [ ] On-call人员已确定?
- [ ] 文档已更新(用户文档、运维文档)?

## 八、特殊情况额外检查

### 如果涉及数据库变更:
- [ ] 备份在迁移前完成?
- [ ] 迁移脚本在Staging上测试过?

- [ ] 第三方变更的监控?

### 如果涉及支付/金融:

- [ ] 异常金额告警?

### 如果涉及用户数据迁移:
- [ ] 已在Staging用真实数据全量演练?
- [ ] 失败的回滚方案?
- [ ] 用户通知?
- [ ] 客服已准备?

【特别提醒:典型上线翻车原因】

我特别警惕以下"看起来没问题但翻车率高"的情况:

1. **小改动综合症**:"只改了一行字"——经常引发CSS连锁反应
2. **赶时间综合症**:为了某个deadline匆忙上线 → bug高发
3. **乐观偏差**:"测试都过了应该没事" → 测试没覆盖到的就翻车
4. **复杂依赖**:涉及3+个外部服务 → 总有一个会出问题
5. **数据迁移**:涉及修改已有数据 → 高危操作
6. **新依赖**:引入团队不熟悉的库 → 隐藏bug
7. **节假日前**:团队不在线时出事 → 灾难

如果出现这些情况,**审查标准要更严**。
```

## 使用流程

### 步骤1:确认前置完成

在使用本提示词前,**必须**完成:
- [ ] 所有CI检查绿
- [ ] 已部署到Staging
- [ ] 已在Staging手动验证
- [ ] 已配置Feature Flag
- [ ] 已准备回滚方案

### 步骤2:收集材料

把所有相关信息整理好:
- 需求文档
- CI最新报告(GitHub Actions页面截图或文字)
- git log / PR diff摘要
- Staging测试报告
- 之前的Code Review报告
- 准备添加的监控

### 步骤3:执行审查

打开新Claude对话,粘贴提示词+所有材料。

### 步骤4:严肃对待结果

报告中:
- 🔴 **必须**解决,不解决不能上线
- 🟠 **强烈建议**解决,无法解决要明确豁免理由
- 🟡 可以上线后处理,但要记录issue

### 步骤5:获得批准

把审查报告交给"AI代码守护者"和"产品负责人":
- 审查报告 + 你的处理计划 → 批准 → 上线
- 不批准 → 修复后重新走流程

---

## 这个提示词为什么有效?

### 1. 全维度覆盖
比单一维度审查更完整,避免"前面都好但忽略了某个角落"。

### 2. 强制结论
"是否可以上线"必须明确表态,不允许模糊。

### 3. 阻断级分类
不是所有问题都是阻断级,但阻断级问题必须解决。

### 4. 特殊情况额外检查
针对支付、数据迁移等高危场景有额外要求。

### 5. 人为因素纳入
考虑发布时间、值守人员等"非技术"因素。

## 业务人员的心态

### 不要害怕"不能上线"的结论

这是体系健康的信号,说明:
- 自动化抓到了真问题
- 节省了一次潜在事故

如果**每次审查都是"可以上线"**,反而要警惕——可能审查不够严格。

### 不要为了上线而妥协

常见的错误心态:
- "已经答应业务方今天上线了..."
- "再等一天就赶不上节奏..."
- "这个问题以后修也行..."

正确心态:
- 上线时间可以推迟,事故无法挽回
- 业务方的真正需求是"功能稳定能用",不是"今天上线"
- "以后修"通常变成"永远不修"

## 实战:典型审查报告

```
# 上线前综合审查报告

## 综合结论
**是否可以上线**: ⚠️ 谨慎(需解决1个🔴问题)

**核心理由**: 整体质量不错,但发现一个安全漏洞必须修复。

## 🔴 阻断级问题

### 问题1: 用户头像URL未验证,可能SSRF
- 位置: `src/features/user/avatar.ts:23`
- 描述: 用户可以提交任意URL作为头像,服务器会fetch该URL生成缩略图。
       恶意用户可提交内网URL(如 http://localhost:6379)进行SSRF攻击。
- 建议: 必须验证URL,只允许公网域名,且封禁内网IP。

## 🟠 严重问题

### 问题1: 没有针对大量订单用户的性能测试
- 描述: 测试用户都是5-10个订单,生产可能有用户1000+订单
- 建议: 准备1000订单测试用户,验证性能。

## 🟡 改进建议

1. 一些组件缺少加载状态
2. 错误提示文案可以更友好
3. 可以添加用户操作引导

## 上线建议

### 必须先做:
1. 修复🔴问题(SSRF漏洞)
2. 解决或豁免🟠问题
3. 重新走完整流程

### 建议发布策略
- 修复🔴后,先灰度1%观察24小时
- 如无异常,逐步推广
- 周二上午发布(避开周末)

### 上线后24小时监控
- API错误率(关注是否>0.5%)
- 头像上传成功率
- 用户反馈

### 立即回滚的标准:
1. API错误率 > 1%
2. 出现新的安全告警
3. 用户大量投诉

## 给业务人员的话

整体准备得不错,但发现了一个安全漏洞必须先修。这个漏洞看起来
"不显眼"但影响严重,如果不修就上线,可能让我们的服务器被
利用攻击其他系统。

预计修复+重新走流程需要 1-2 天,建议把上线时间推迟到 X月X日。
```

## 总结

本提示词是**质量保障的最后一道关**。

业务人员可能因为:
- 时间压力
- 业务催促
- 自我感觉良好

而想跳过这一步。**绝不能跳过**。

每一次严格的审查,都是对自己、对用户、对公司的负责。
