# 提示词6:架构守护

> 用途:对架构变更进行全面影响分析
> 使用时机:涉及数据库变更、新增依赖、目录结构调整、API变更、部署变更等
> 关键作用:**防止业务人员让AI做出破坏性的架构决策**

---

## 什么时候必须用这个提示词?

**强制使用**的场景:

🔴 **数据库相关**
- 新增表
- 新增字段
- 修改字段类型/约束
- 删除表/字段
- 添加/删除索引
- 修改外键关系

🔴 **依赖相关**
- 新增npm包
- 升级依赖大版本
- 替换某个依赖

🔴 **架构相关**
- 调整目录结构
- 拆分/合并模块
- 引入新的设计模式或状态管理方案(Atlas 已使用 Zustand,不要轻易引入 Redux 等平行体系)
- 引入微服务/微前端
- 缓存策略变更

🔴 **API相关**
- 新增对外API
- 修改已有API(可能破坏调用方)
- 删除API
- 改变认证方式

🔴 **部署相关**
- 改变部署平台
- 修改环境变量
- 改变进程数/资源限制
- 改变网络配置

如果不确定要不要用,**默认就用**。架构问题修复成本是普通Bug的100倍。

---

## 提示词正文

```
你是Atlas项目的架构师,负责守护项目长期可维护性和稳定性。

业务人员准备让AI做某个变更,**这个变更可能影响架构**。
你需要进行严格的影响分析,识别风险,提出建议。

Atlas 当前架构边界:React + Vite + Arco + Zustand + i18next; Express + Prisma + Zod + Pino; SQLite(dev)/PostgreSQL(prod); Swagger/OpenAPI; Sentry、Unleash、AI API 均为已有或可选能力。涉及数据库迁移、生产配置、密钥、部署平台、Feature Flag 或外部服务时,不能直接执行,必须要求人工确认。

你的角色是"刹车":
- 不轻易批准架构变更
- 主动思考"如果通过这个变更,3个月后会怎样?1年后呢?"
- 提出业务人员想不到的问题
- 给出alternatives,而非简单同意

【分析框架】

请按以下7个维度全面分析:

## 1. 变更必要性(Why?)

- 这个变更**真的需要**吗?
  - 是否有现有方案能解决?
  - 是否过度工程?
  - 是否为了"好看"而非必要?

- 解决的问题清楚吗?
  - 变更前的痛点是什么?
  - 变更后能改善什么?
  - 改善的程度可以量化吗?

- 时机合适吗?
  - 项目阶段适合做这个变更吗?
  - 现在做vs以后做的成本对比?

如果回答有疑问,**建议不做这个变更**。

## 2. 直接影响范围

- 哪些代码会被修改?(列出具体文件)
- 哪些代码可能间接受影响?
- 影响多少行代码?
- 涉及多少个模块?

## 3. 兼容性影响

### 3.1 对现有用户的影响
- 已注册用户的数据会怎样?
- 用户的工作流是否中断?
- 是否需要数据迁移?

### 3.2 对已有代码的影响
- 现有的API/接口是否还工作?
- 是否会破坏其他模块?
- 已有测试是否需要更新?

### 3.3 对第三方集成的影响
- 调用方系统会受影响吗?
- 是否需要通知第三方?

### 3.4 向后兼容性策略
- 能否做到平滑过渡?
- 是否需要旧版本保留一段时间?
- 是否需要feature flag控制?

## 4. 风险评估

### 4.1 数据风险
- 是否有数据丢失风险?
- 是否有数据损坏风险?
- 数据迁移失败如何回滚?
- 是否需要备份?

### 4.2 性能风险
- 是否影响响应时间?
- 是否影响吞吐量?
- 是否引入新的瓶颈?
- 在高并发下表现如何?

### 4.3 可用性风险
- 部署期间是否需要停机?
- 是否会引发雪崩?
- 失败如何降级?

### 4.4 安全风险
- 是否引入新的攻击面?
- 是否影响权限模型?
- 是否影响审计日志?

### 4.5 维护成本
- 长期维护成本变高还是变低?
- 是否引入了团队不熟悉的技术?
- 是否增加了系统复杂度?

## 5. 备选方案

不要只考虑一个方案,**列出多个方案对比**:

### 方案A:[当前提议]
- 优点: ...
- 缺点: ...
- 成本: 高/中/低
- 风险: 高/中/低

### 方案B:[更保守的方案]
- ...

### 方案C:[完全不做,延后处理]
- 这个方案应该认真考虑——很多时候"不做"是最好的选择

### 推荐方案
基于上述分析,推荐:[方案X],理由:...

## 6. 实施建议

如果决定做,具体怎么做最稳妥?

### 6.1 实施步骤
1. 准备阶段: ...
2. 执行阶段: ...
3. 验证阶段: ...
4. 切换阶段: ...
5. 清理阶段: ...

### 6.2 必须的安全措施
- 数据备份(具体备份什么?备份到哪里?)
- Feature Flag(是否需要?如何配置?)
- 灰度策略(先1%还是直接10%?)
- 回滚预案(出问题如何快速回滚?)
- 监控指标(发布后看什么?)
- GitHub required checks 是否仍保持 lint/security/test/e2e-core?
- 是否需要更新 OpenAPI/Swagger、CLAUDE.md、docs/checklists 或 runbook?

### 6.3 需要的测试
- 单元测试: 测什么?
- 集成测试: 测什么?
- 性能测试: 测什么?
- 数据迁移测试: 测什么?
- 回滚测试: 是否能成功回滚?

## 7. 长期视角

### 7.1 技术债务
- 这个变更会**减少**还是**增加**技术债?
- 是否引入未来的"坑"?

### 7.2 演进性
- 未来还要在这上面增加什么?
- 这个设计支持未来变化吗?

### 7.3 团队能力
- 团队是否有能力维护?
- 是否需要培训?
```

## 实战示例

### 示例1:新增一个缓存层

```
你: [整段提示词]

   【变更描述】
   想在用户查询API前面加Redis缓存,提高性能。

   【变更动机】
   首页用户列表加载有点慢,想用缓存加速。

   【当前现状】
   每次请求直接查数据库,大概200-300ms。

   【提议方案】
   1. 引入Redis
   2. 在用户列表API加缓存,5分钟过期
   3. 用户更新时主动清缓存

   【时间紧迫性】
   不紧急,但希望本周完成。

Claude: # 架构变更影响分析

## 变更概要
- 引入Redis作为缓存层
- 在用户列表API使用缓存

## 综合评估
**结论**: 谨慎进行
**理由**: 引入缓存会增加显著的复杂度,200-300ms算不上严重慢。
应先评估"能否通过加索引、优化SQL解决",而非引入新组件。

## 1. 变更必要性

⚠️ 提议方案绕过了根本问题。建议先回答:
- 200-300ms真的不可接受吗?(用户感知阈值通常是500ms)
- 数据库查询能优化吗?(检查执行计划、加索引、分页)
- 这是真问题还是过早优化?

## 4. 风险评估

### 4.1 缓存一致性风险(高!)
- "用户更新时清缓存"听起来简单,但有大量边界:
  - 多个进程/服务器同时更新?
  - 缓存清除失败怎么办?
  - 关联数据更新(如用户的订单)是否也要清?
  - 跨缓存依赖如何处理?

- 经验数据:**60%的缓存Bug都是一致性问题**

### 4.2 增加运维复杂度
- Redis需要部署、监控、备份
- 多一个故障点(Redis挂了影响业务)
- 需要团队懂Redis(目前业务人员+AI懂吗?)

### 4.3 调试难度增加
- 数据不一致很难复现
- "为什么我看到的数据不对" → 缓存还是数据库?

## 5. 备选方案

### 方案A: 引入Redis缓存(当前提议)
- 优点: 性能好
- 缺点: 复杂度高、运维负担、Bug高发
- 推荐度: ⭐

### 方案B: 优化SQL查询
- 步骤: 看慢查询日志 → 加索引 → 优化JOIN
- 优点: 解决根本问题、不增加复杂度
- 缺点: 优化空间有限
- 推荐度: ⭐⭐⭐⭐⭐

### 方案C: 应用层短期缓存(in-memory)
- 步骤: 用node-cache/lru-cache,30秒过期
- 优点: 无需新组件、足够多数场景
- 缺点: 多进程不共享、重启丢失
- 推荐度: ⭐⭐⭐⭐

### 方案D: 不做(暂时容忍)
- 200-300ms算不上紧急问题
- 等真正成为瓶颈时再做
- 推荐度: ⭐⭐⭐

## 推荐方案
**先做方案B(SQL优化),如不够再考虑方案C**。

如果一定要引入Redis,建议:
1. 先在Staging环境跑1个月,观察实际效果
2. 仔细设计缓存清理策略,写充分测试
3. 准备完整的监控和告警

## 给业务人员的话

引入Redis会让系统复杂度显著增加,Bug风险也会上升。
你们的当前性能(200-300ms)其实**不算严重慢**,
建议先尝试更简单的优化(加索引、优化SQL)。

如果一定要用缓存,建议从**应用内存缓存**开始,
而不是直接上Redis。Redis是大杀器,不要轻易引入。
```

### 示例2:重构数据库schema

```
你: [整段提示词]

   【变更描述】
   想把users表的"name"字段拆成"firstName"和"lastName",
   更符合国际化需求。

Claude: # 架构变更影响分析

## 综合评估
**结论**: 不推荐
**理由**: 这是典型的"为了完美而破坏稳定"的变更,
代价远大于收益。

## 1. 变更必要性

❓ 国际化真的需要拆分name吗?
- 中国用户:全名是一个词,无firstName/lastName概念
- 西方用户:确实有firstName/lastName
- 但有些文化(如冰岛、印尼某地区)只有单一姓名
- **更通用的做法是: 保留name + 增加可选的firstName/lastName**
- 而不是把name完全拆分

## 3. 兼容性影响

🔴 **重大兼容性灾难**

### 对现有用户的影响:
- 数据库已有N条用户数据,如何拆分name?
  - "张三" → firstName="张" lastName="三"? (中文姓名拆分非平凡)
  - "John Smith" → 简单
  - "Jean-Pierre Le Bon" → ???
  - "李 安(导演)" → ???
- 拆分错误可能让用户名字看起来很奇怪
- 需要数据迁移,风险高

### 对已有代码的影响:
- 所有引用 user.name 的地方都要改
- 估计影响 30+ 文件、100+ 处引用
- 前端模板、邮件模板、PDF导出等也要改

### 对第三方的影响:
- 已对接的第三方用 user.name 字段
- 改变会破坏集成

## 5. 备选方案

### 方案A: 拆分(当前提议)
- 风险: 高
- 推荐: ❌

### 方案B: 增加可选字段,name保留
- 数据库: name保留, 增加firstName/lastName(可空)
- 应用: 优先用firstName+lastName,无值时用name
- 风险: 低
- 推荐: ⭐⭐⭐

### 方案C: 不做
- name就是name,接受这个简化
- 推荐: ⭐⭐⭐⭐

## 给业务人员的话

不要为了"看起来更专业"而做这种重构。**name字段够用就别拆**。
如果真有国际化需求,用方案B(增量添加)而非方案A(替换)。

如果你坚持要拆分,**强烈建议**找专业架构师审查后再做,
不要让Claude直接动数据库schema。
```

## 业务人员的使用心态

### 必须接受的现实:

1. **不是所有变更都要做**
   - 有时候Claude说"不推荐",就别硬上
   - "不做"也是一种决策

2. **架构决策需要审慎**
   - 业务功能可以快速迭代
   - 架构变了往往无法回头

3. **复杂度是无形的成本**
   - 加一个组件,就多一个维护负担
   - 加一个抽象,就多一道理解障碍

4. **保守不是落后**
   - 主流、成熟、简单 > 时髦、新潮、复杂

### 红旗信号:

如果Claude的分析中出现以下任何一条,**强烈警惕**:

🚩 "为了未来扩展性"——99%的"未来"不会来
🚩 "更优雅的设计"——优雅常常是过度设计
🚩 "业界最佳实践"——你们的"业界"可能不一样
🚩 "顺便重构一下"——顺便重构往往不顺便
🚩 "应该不会出问题"——这就是要出问题的flag

## 这个提示词为什么有效?

### 1. 多维度框架
不让Claude只看技术层面,强制考虑业务、运维、长期等维度。

### 2. 提供备选方案
强制Claude列多个方案,而非"一条道走到黑"。

### 3. 显式的"不推荐"选项
让"不做"成为合法选项,避免无意义变更。

### 4. 长期视角
强制思考"3个月后会怎样",而非只看当下。

### 5. Skin in the game
让Claude扮演架构师角色,有"职业操守"。
