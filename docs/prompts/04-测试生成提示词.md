# 提示词4:测试生成

> 用途:让Claude为已有代码生成全面的测试
> 使用时机:代码生成后,或为遗留代码补测试
> 关键产出:可以直接运行的、覆盖全面的、高质量的测试代码

---

## 提示词正文

```
你是测试领域的专家,精通:
- TDD/BDD开发方法
- 测试金字塔(单元/集成/E2E)
- 各种边界场景设计
- Mocking与测试隔离
- 性能与安全测试

你的任务是为给定的代码生成全面、高质量、可直接运行的测试。

Atlas 测试基线:
- 单元测试:Vitest
- 后端集成测试:Vitest + Supertest + Prisma,新 worktree 先执行 `cd server && npx prisma generate`
- E2E:Playwright,核心门禁命令是 `npm run test:e2e:core`
- 可访问性/视觉:Playwright + axe-core,通过 `npm run test:e2e:a11y` 和 `npm run test:e2e:visual`
- 不要臆造 Fastify `app.inject`、Jest、Testcontainers 或邮箱字段;Atlas User 模型没有 email/phone,登录使用 username/password,联系人使用 realName/canLogin。

【绝对原则】

1. **测试必须有意义**
   - 不为覆盖率而写测试,每个测试都要有明确目的
   - 不写"测试setter的setter测试"这类无意义测试
   - 优先测试"业务规则"和"边界情况",而非"是否调用了某方法"

2. **测试必须独立**
   - 任何两个测试可以独立运行,不依赖执行顺序
   - 每个测试自己setup/teardown
   - 不共享可变状态

3. **测试必须快速**
   - 单元测试每个 < 100ms
   - 不用 sleep / setTimeout / wait
   - 用 mocks/fakes 隔离慢的依赖(数据库、网络)
   - 集成测试可以慢一点,但要并行

4. **测试必须可读**
   - 测试名清楚说明:在什么前置下,做什么操作,期望什么结果
   - 用AAA模式:Arrange(准备) - Act(执行) - Assert(断言)
   - 测试代码也要遵守命名规范,避免magic value

5. **测试必须健壮**
   - 不依赖具体时间(用固定测试时间)
   - 不依赖具体随机数(seed或mock)
   - 不依赖网络(mock外部依赖)
   - 不依赖文件系统(用内存/临时目录)

---

【测试覆盖清单】请确保以下场景都被覆盖:

## 1. 正常路径(Happy Path)
- 典型输入返回预期输出
- 主流程完整跑通

## 2. 边界值
- 空值: null, undefined, '', [], {}
- 零值: 0, 0.0, false
- 负值: -1, 负数
- 极值: MAX_INT, MIN_INT, 超长字符串(1万字符), 大数组(1万元素)
- 浮点精度: 0.1+0.2 != 0.3
- 类型边界: NaN, Infinity, -Infinity

## 3. 特殊输入
- emoji: '😀🎉'
- Unicode: '中文', 'العربية'
- 控制字符: '\n\t\r'
- 引号: "'", '"'
- HTML/JS: '<script>alert(1)</script>'
- SQL: "' OR '1'='1"
- 路径遍历: '../../../etc/passwd'

## 4. 错误路径
- 每个抛错分支都有测试
- 每种错误类型都有测试
- 错误消息符合预期

## 5. 异常依赖
- 数据库连接失败
- 数据库返回空
- 数据库返回脏数据
- 外部API超时
- 外部API返回错误
- 外部API返回意外格式

## 6. 并发场景(如适用)
- 同时多个请求修改同一资源
- 重复提交
- 竞态条件

## 7. 时间相关
- 不同时区
- 跨时区
- 闰年
- 月底/年底
- 历史日期 vs 未来日期

## 8. 安全相关(关键!)
- SQL注入测试
- XSS测试
- 越权测试(用户A不能访问用户B的数据)
- 未认证访问
- 权限不足

## 9. 性能相关(适用时)
- 大数据量(10万条)的处理时间
- 并发处理能力

## 10. 集成场景(API测试)
- 完整请求-响应链路
- 中间件链(认证、限流、日志)
- 数据库事务
- 缓存一致性(仅当改动涉及现有缓存或新增缓存方案)

【测试质量要求】

### 命名规范

测试名清晰说明三要素:
```
✅ 好的测试名:
- "登录_使用正确凭证_应返回Token"
- "用户列表_未登录_应返回401"
- "活动创建_缺少项目权限_应返回403"
- "should return 401 when user is not authenticated"
- "should reject username when format is invalid"

❌ 不好的测试名:
- "test1"
- "登录测试"
- "login"
- "should work"
```

### AAA模式

```typescript
test('订单创建_库存不足_应抛出InsufficientStockError', async () => {
  // Arrange - 准备测试数据
  const product = await createTestProduct({ stock: 0 });
  const user = await createTestUser();

  // Act - 执行被测代码
  const action = () => createOrder(user.id, [{ productId: product.id, qty: 1 }]);

  // Assert - 验证结果
  await expect(action).rejects.toThrow(InsufficientStockError);
});
```

### 工厂模式生成数据

```typescript
// ✅ 好:用工厂函数,默认值合理,可覆盖
function createTestUser(overrides = {}) {
  return {
    id: 'usr_' + randomString(),
    username: `test_${Date.now()}`,
    realName: '测试用户',
    canLogin: true,
    status: 'ACTIVE',
    createdAt: new Date('2024-01-01'),
    ...overrides
  };
}

// 使用
const disabledUser = createTestUser({ status: 'DISABLED' });
const oldUser = createTestUser({ createdAt: new Date('2020-01-01') });

// ❌ 不好:每个测试都手写完整对象
test('...', () => {
  const user = {
    id: 'u1', username: 'u1', realName: '', canLogin: true, /* 还有字段 */
  };
  // ...
});
```

### 不依赖测试顺序

```typescript
// ✅ 好:每个测试独立创建数据
test('test 1', async () => {
  const user = await createTestUser();
  // ...
  await deleteUser(user.id);
});

test('test 2', async () => {
  const user = await createTestUser();
  // ...
  await deleteUser(user.id);
});

// ❌ 不好:测试间共享数据
let user;
beforeAll(async () => {
  user = await createTestUser();  // 如果test1把user删了,test2会失败
});
```

### Mock外部依赖

```typescript
// ✅ Mock外部HTTP服务,例如AI能力
test('AI生成摘要_接口超时_应返回降级错误', async () => {
  // Arrange
  vi.spyOn(aiClient, 'generateSummary').mockImplementation(
    () => new Promise((_, reject) =>
      setTimeout(() => reject(new TimeoutError()), 0)
    )
  );

  // Act & Assert
  await expect(generateWeeklySummary(report)).rejects.toThrow(AiTimeoutError);
});

// ✅ 后端集成测试使用项目已有 Prisma/Vitest/Supertest 测试基础设施。
// 如果需要 Docker 测试环境,优先使用根目录 `npm run test:env:*` 脚本。
```

## 使用场景

### 场景1:新功能配套测试

刚生成新代码后:
```
工作流程:
1. 用《代码生成提示词》生成代码
2. 用本提示词生成配套测试
3. 用《代码审查提示词》审查代码+测试
4. 修改完善
5. 提交PR
```

### 场景2:为遗留代码补测试

如果Atlas项目有"暗代码"(没有测试的旧代码):
```
1. 拿到代码
2. 用本提示词生成测试
3. 运行测试,如果有失败:
   a. 是bug → 修代码
   b. 是测试理解错了 → 修测试(同时记录这个误解)
4. 持续直到达标
```

发现Bug时,**先写复现Bug的测试**,再修代码:

## 测试代码示例

### 示例:用户登录测试

```typescript
// server/src/__tests__/integration/auth-login.test.ts

import request from 'supertest';
import { beforeEach, describe, expect, it } from 'vitest';
import bcrypt from 'bcrypt';
import app from '../../app';
import { prisma } from '../../utils/prisma';

describe('POST /api/auth/login', () => {
  beforeEach(async () => {
    await prisma.user.deleteMany({ where: { username: { startsWith: 'test_login_' } } });
  });

  describe('正常场景', () => {
    it('使用正确用户名和密码_应返回Token和用户信息', async () => {
      // Arrange
      const password = 'SecurePass123!'; // pragma: allowlist secret
      const user = await prisma.user.create({
        data: {
          username: 'test_login_success',
          realName: '测试登录用户',
          canLogin: true,
          status: 'ACTIVE',
          password: await bcrypt.hash(password, 12),
        },
      });

      // Act
      const response = await request(app)
        .post('/api/auth/login')
        .send({ username: 'test_login_success', password });

      // Assert
      expect(response.status).toBe(200);
      expect(response.body).toMatchObject({
        token: expect.stringMatching(/^eyJ/),
        user: {
          id: user.id,
          username: 'test_login_success',
          realName: '测试登录用户',
        },
      });
      expect(JSON.stringify(response.body)).not.toContain('password');
    });
  });

  describe('输入验证', () => {
    it('用户名为空_应返回400', async () => {
      const response = await request(app)
        .post('/api/auth/login')
        .send({ username: '', password: 'Pass1234' }); // pragma: allowlist secret

      expect(response.status).toBe(400);
    });

    it('请求体为空_应返回400', async () => {
      const response = await request(app).post('/api/auth/login').send({});

      expect(response.status).toBe(400);
    });
  });

  describe('安全测试', () => {
    it('SQL注入字符串_应作为普通用户名处理', async () => {
      const response = await request(app)
        .post('/api/auth/login')
        .send({ username: "admin'--", password: 'anything' }); // pragma: allowlist secret

      expect(response.status).toBe(401);
    });

    it('用户不存在_应返回相同的错误信息_防止用户枚举', async () => {
      await prisma.user.create({
        data: {
          username: 'test_login_exists',
          realName: '已存在用户',
          canLogin: true,
          status: 'ACTIVE',
          password: await bcrypt.hash('CorrectPass123', 12),
        },
      });

      const response1 = await request(app)
        .post('/api/auth/login')
        .send({ username: 'test_login_missing', password: 'Pass1234' }); // pragma: allowlist secret
      const response2 = await request(app)
        .post('/api/auth/login')
        .send({ username: 'test_login_exists', password: 'WrongPass123' }); // pragma: allowlist secret

      expect(response1.body.error).toEqual(response2.body.error);
      expect(response1.status).toBe(response2.status);
    });

    it('canLogin为false的联系人_不能登录', async () => {
      await prisma.user.create({
        data: {
          username: 'test_login_contact',
          realName: '仅联系人',
          canLogin: false,
          status: 'ACTIVE',
        },
      });

      const response = await request(app)
        .post('/api/auth/login')
        .send({ username: 'test_login_contact', password: 'anything' }); // pragma: allowlist secret

      expect(response.status).toBe(401);
    });
  });
});
```

看到了吗?一个登录API至少要覆盖成功、输入验证、注入防护、用户枚举、仅联系人不可登录、敏感字段不泄露等关键路径。实际代码请以仓库现有测试 helper 和路由返回格式为准,不要照抄示例里的导入路径。

## 这个提示词为什么有效?

### 1. 详尽场景清单
不依赖Claude"想到什么测什么",而是有清晰的checklist。

### 2. 强制思考边界
特别强调边界值、特殊输入、安全场景——这些是bug高发区。

### 3. 高质量代码模式
通过强调AAA、工厂模式、独立性,提高测试代码本身的质量。

### 4. 现实主义
要求Claude诚实指出"未能覆盖的"和"建议手动测试的",不夸大。

### 5. 与Mutation测试呼应
提前预测哪些mutation难kill,引导测试更严谨。
