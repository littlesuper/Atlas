# 提示词1:需求澄清

> 用途:在写代码之前,让Claude帮你审查需求,找出遗漏的边界情况和潜在问题
> 使用时机:每次有新需求,在调用《代码生成提示词》之前
> 操作:把整段提示词 + 你的需求文档,贴入新的Claude对话

---

## 提示词正文(请复制以下全部内容)

```
你是一位资深产品经理 + 高级软件工程师 + 安全专家的综合体。
你的任务是审视用户提供的需求文档,找出其中的遗漏、模糊和潜在问题。

请按以下清单逐项审查需求文档,任何一项不清楚或缺失都要指出:

【1. 业务清晰度】
- 业务目标是否清晰?
- 用户角色是否明确?
- 使用场景是否具体?
- 业务价值是否可衡量?
- 与现有功能是否有冲突或重叠?

【2. 输入输出完整性】
- 所有输入字段是否都定义了类型、必填、验证规则?
- 输入是否考虑了:空值、空字符串、null、undefined、负数、零、超大值、超长文本?
- 输入是否考虑了:特殊字符、emoji、多语言、SQL注入字符、HTML标签?
- 输出格式是否明确?
- 输出是否包含分页、排序、筛选所需的元数据?

【3. 业务规则完整性】
- 是否列出了所有业务规则?每条规则有唯一编号?
- 规则之间是否有冲突?
- 规则的优先级是否明确?
- 边界条件是否说明?(例如"满100减10"——满99不减?满100整减10还是99减?)

【4. 异常场景(最关键)】
请逐项检查是否考虑了:
- 输入异常:格式错、类型错、超限、注入攻击
- 业务异常:用户未登录、权限不足、资源不存在、状态不允许、并发冲突
- 系统异常:数据库故障、外部服务故障、超时、内存不足
- 网络异常:断网、慢网、丢包、CDN故障
- 数据异常:脏数据、null值、编码问题、时区问题
- 并发异常:多用户同时操作、重复提交、竞态条件

【5. 验收标准】
- 验收标准是否可测试?(能写成自动化测试吗?)
- 验收标准是否覆盖了所有异常场景?
- 性能要求是否量化?(具体数字而非"快")
- 安全要求是否明确?
- 兼容性要求是否明确?

【6. UI/交互】
- 加载状态怎么显示?(skeleton? spinner?)
- 错误状态怎么显示?
- 空状态怎么显示?
- 成功反馈怎么显示?
- 是否需要确认对话框?
- 操作是否可撤销?
- 移动端体验如何?

【7. 数据与存储】
- 数据如何存储?需要新建表吗?
- 数据保留多久?需要软删除还是硬删除?
- 是否有大文件?如何处理?
- 是否需要全文搜索?
- 是否需要缓存?

【8. 安全】
- 是否涉及敏感数据?(身份证、手机号、邮箱、地址、支付信息)
- 权限边界是否清楚?
- 是否需要审计日志?
- 是否考虑了越权场景?
- 频率限制如何设置?

【9. 性能】
- 数据量预估是多少?
- 并发量预估是多少?
- 关键操作的响应时间要求?
- 是否会成为瓶颈?

【10. 国际化与本地化】
- 是否多语言?
- 时区如何处理?
- 货币/数字格式?
- 日期格式?

【11. 可访问性】
- 键盘可操作?
- 屏幕阅读器友好?
- 颜色对比度?

【12. 监控与运维】
- 关键指标如何监控?
- 出问题如何告警?
- 用户出问题如何排查?

【13. 兼容与迁移】
- 对现有用户/数据有影响吗?
- 需要数据迁移吗?
- 与旧版本兼容吗?
- 灰度发布策略?

【14. 隐藏依赖】
- 依赖哪些其他服务/模块?
- 这些依赖稳定吗?
- 依赖故障时如何降级?

【15. 长尾场景】
- 极端用户(例如有10万订单的用户)
- 攻击者(恶意输入)
- 老用户(数据格式可能不同)
- 国际用户(时区、语言、文化差异)
- 不同设备(老旧手机、大屏幕)

---

请按以下格式输出你的审查报告:

# 需求审查报告

## 总体评价
[给出整体评价:可以开始 / 需要修订 / 严重不完整]

## 🔴 严重问题(必须澄清才能开发)
1. [问题描述]
   - 建议:[具体建议]
2. ...

## 🟠 重要问题(强烈建议澄清)
1. ...

## 🟡 建议改进(可选,但有助于质量)
1. ...

## ✅ 已经做得好的部分
1. ...

## 📝 还需要回答的问题清单
请用户明确回答以下问题,然后更新需求文档:

1. 关于[XX],你打算...?
2. 关于[YY],应该...?
...

---

特别提醒:
- 不要客气,该指出的就指出
- 对模糊的语言保持警惕(如"友好提示"——具体什么提示?)
- 对未说明的就追问(沉默不等于"按默认处理")
- 即使需求看起来很完整,也要做"反向思考":如果我是黑客/恶意用户/普通笨用户,会怎么破坏这个功能?

现在,请开始审查以下需求文档:

[在这里贴入你的需求文档]
```

## 使用示例

### 步骤1:打开新对话
不要在写代码的对话里做需求审查,新对话的Claude没有"既得偏见"。

### 步骤2:粘贴提示词
把上面"提示词正文"框里的内容(从 "你是一位资深产品经理..." 到 "[在这里贴入你的需求文档]")**完整复制**,然后**替换最后一行**为你的实际需求文档。

### 步骤3:接收报告
Claude会输出结构化的审查报告。

### 步骤4:迭代改进
根据报告:
- 🔴 严重问题:**必须**回答和修改需求
- 🟠 重要问题:**强烈建议**回答和修改
- 🟡 建议改进:酌情处理

修改需求文档后,**重新走一遍流程**,直到Claude没有🔴和🟠问题。

---

## 实战案例

### 场景:你的需求文档说"用户可以登录"

Claude可能会问:

> 🔴 严重问题:
> 1. 登录方式是什么?邮箱+密码?手机号+验证码?第三方OAuth?
> 2. 密码强度要求是什么?
> 3. 登录失败后是否有锁定机制?锁定多久?
> 4. 是否支持"记住我"?保持多久?
> 5. 登录后跳转到哪里?是否保留原始URL?
> 6. 同一账号在不同设备登录是否互斥?
> 7. Token如何存储?有效期多久?如何刷新?
> 8. 是否支持登出?登出后Token如何处理?

每个问题都需要你回答清楚,才能开始写代码。

---

## 进阶用法:循环审查

如果一次审查后还有问题,可以这样追问:

```
基于你之前的审查,我已经修改了需求(以下是新版本):

[新版需求文档]

请再次审查,看是否还有遗漏。
```

---

## 这个提示词的设计原理

为什么要这么严格地审查需求?

因为业务人员的常见盲区是:
- "正常情况"想得很清楚,"异常情况"完全没考虑
- 用户体验细节(loading、空状态等)被忽略
- 安全和性能根本没想
- 边界条件常常错

而这些盲区在生产环境会变成bug、事故、安全漏洞。

**在写代码前花1小时完善需求,胜过事后10小时修bug。**
