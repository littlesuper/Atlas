# Atlas 第一次应急演练方案

> 用途：Week 7 Day 5 第一次应急演练。本文档只准备演练计划，不直接制造 staging / production 故障。真实演练必须由发布负责人确认环境、时间窗和参与人后执行。

## 演练目标

- 验证团队是否能在 5 分钟内识别事故、定级并启动统一响应。
- 验证 `docs/qa/incident-response.md`、`docs/qa/degradation-switches.md`、`docs/qa/rollback-runbook.md` 是否可按步骤执行。
- 测量响应时间、止血时间、恢复时间和沟通节奏。
- 找出联系人、权限、工具入口、Runbook、监控和测试上的缺口。

## 演练边界

| 项目 | 要求 |
| ---- | ---- |
| 环境 | 首选 staging；没有明确授权前只做桌面推演或本地演练 |
| 数据 | 不使用生产数据，不执行破坏性数据库命令 |
| 密钥 | 不新增、不修改、不暴露任何密钥 |
| 生产 | 不对 production 注入故障 |
| 回滚 | 真实 staging 回滚需要发布负责人现场确认 |
| 通知 | 即使是不预告演练，也必须有演练安全负责人知道并能随时叫停 |

## 角色分工

| 角色 | 职责 | 人员 |
| ---- | ---- | ---- |
| 演练安全负责人 | 确认环境、控制开始/停止、叫停风险动作 | 待定 |
| AI 守护人 | 准备场景、记录时间线、辅助查文档 | Codex |
| 值班人 | 按应急清单响应、定级、执行止血动作 | 待定 |
| 发布负责人 | 判断是否降级/回滚、执行发布相关动作 | 待定 |
| 业务负责人 | 判断用户影响、确认恢复标准 | 待定 |
| 观察员 | 只记录，不提示，复盘时反馈卡点 | 待定 |

## 推荐演练场景

### 场景 A：可选模块 500，优先 Feature Flag 降级

目标：验证“先关 Flag，后追因”是否能执行。

注入方式候选：

- 在 staging 临时让一个非核心可选模块 API 返回 500。
- 推荐模块：AI 辅助、企微登录、项目模板、风险总览、负载看板。
- 不选择登录基础流程、项目详情、活动编辑等核心路径。

期望响应：

1. 值班人发现异常并定级为 P1/P2。
2. 查 `docs/qa/degradation-switches.md` 找到对应 Flag。
3. 发布负责人将 rollout 降到 0% 或关闭 Flag。
4. 验证核心路径恢复。
5. 记录从发现到止血的耗时。

### 场景 B：新版本核心路径异常，触发回滚决策

目标：验证“发布后 1 小时内核心路径异常，优先回滚”的判断。

注入方式候选：

- 在 staging 部署一个已知会破坏项目列表或活动列表的测试分支。
- 不修改数据库 schema，不触发数据破坏。

期望响应：

1. 值班人识别核心路径受影响并定级为 P0/P1。
2. 发布负责人确认 Feature Flag 无法隔离。
3. 按 `docs/qa/rollback-runbook.md` 执行代码回滚。
4. 验证 `/api/health`、登录、项目列表、活动列表。
5. 记录回滚总耗时，目标小于 5 分钟。

### 场景 C：第三方依赖异常，验证降级和沟通

目标：验证外部依赖故障时是否能避免扩大影响。

注入方式候选：

- 在 staging 模拟 AI API 或企微 OAuth 超时。
- 不修改真实第三方配置，不写入生产密钥。

期望响应：

1. 判断异常只影响可选依赖。
2. 关闭对应功能 Flag。
3. 用公告模板准备内部/外部说明。
4. 验证账号密码登录、核心项目管理流程不受影响。

## 推荐第一次选择

第一次建议选 **场景 A：可选模块 500，优先 Feature Flag 降级**。

原因：

- 风险最低，不碰数据库。
- 能验证 Day 1-4 已落地的 Feature Flag、降级清单、事故清单和沟通模板。
- 如果团队卡住，可以立即叫停并恢复 Flag。

## 执行流程

### 演练前 15 分钟

- [ ] 演练安全负责人确认 staging URL。
- [ ] 确认 staging 数据可丢弃或已备份。
- [ ] 确认当前 `main` CI 成功。
- [ ] 确认 `docs/qa/incident-response.md` 可访问。
- [ ] 确认 `docs/qa/degradation-switches.md` 可访问。
- [ ] 确认 `docs/qa/public-announcement-templates.md` 可访问。
- [ ] 准备 `docs/qa/reports/emergency-drill-template.md` 的副本作为记录。

### 演练开始

- [ ] 演练安全负责人记录开始时间。
- [ ] AI 守护人或指定人员注入已批准的 staging 故障。
- [ ] 观察员记录团队第一次发现时间。
- [ ] 值班人按应急清单定级。
- [ ] 发布负责人选择降级或回滚。
- [ ] 业务负责人确认恢复验收标准。

### 演练恢复

- [ ] 关闭故障注入。
- [ ] 恢复 Feature Flag / 回滚到稳定版本。
- [ ] 验证 `/api/health`。
- [ ] 验证管理员登录。
- [ ] 验证项目列表。
- [ ] 验证活动列表。
- [ ] 记录恢复完成时间。

### 演练复盘

- [ ] 计算发现耗时、响应耗时、止血耗时、恢复耗时。
- [ ] 标记卡点：人、权限、文档、工具、监控、测试。
- [ ] 形成 3 个以内最高优先级改进行动。
- [ ] 把改动补回 Runbook、清单或 CI。

## 叫停条件

出现任一情况必须立即叫停：

- 注入动作可能影响 production。
- 需要修改密钥、生产配置或数据库迁移。
- staging 数据不可恢复，且演练动作可能造成数据破坏。
- 团队无法确认当前环境。
- 事故沟通已经误发给真实客户或外部渠道。

## 通过标准

- 从故障出现到首次响应小于 5 分钟。
- 从首次响应到止血动作小于 10 分钟。
- 如果执行回滚，代码回滚小于 5 分钟。
- 应急时间线完整。
- 至少产出 1 条可执行改进行动。
- 没有影响 production。
