# Atlas 质量保障小白手册

这份手册给不写代码或刚开始和 AI 协作写代码的 Atlas 团队成员使用。目标不是让每个人都成为工程师,而是让每个改动都有清楚的证据链:需求说清楚、代码被审查、测试跑过、上线可回滚。

## 一句话原则

先写清楚,再让 AI 写;先让另一个 AI 审查,再提 PR;先让 CI 全绿,再合并;先有回滚方案,再上线。

## 一次需求的标准路径

| 阶段 | 你要做什么 | 产物 |
| --- | --- | --- |
| 1. 写需求 | 用需求模板说明目标、角色、流程、异常、验收标准 | `docs/requirements/` 下的需求文档 |
| 2. 澄清需求 | 使用 `01-需求澄清提示词.md` 找缺口 | 问题清单和修订后的需求 |
| 3. 生成代码 | 使用 `02-代码生成提示词.md`,同时贴 CLAUDE.md 和相关代码 | 可审查的代码改动 |
| 4. 生成测试 | 使用 `04-测试生成提示词.md` | Vitest/Supertest/Playwright 测试 |
| 5. 审查代码 | 使用 `03-代码审查提示词.md`,最好开新对话 | 阻断/严重/建议问题清单 |
| 6. 本地验证 | 跑 lint、test、必要的 E2E | 命令输出或截图记录 |
| 7. 提 PR | PR 描述写清楚改了什么、验证了什么、风险是什么 | GitHub PR |
| 8. 等 CI | required checks 必须全绿 | `lint`、`security`、`test`、`e2e-core` |
| 9. 合并 | 由守护人按保护规则合并 | main 分支 CI 仍通过 |
| 10. 上线前检查 | 使用上线前检查清单和提示词 | 发布结论、回滚标准、观察项 |

## 每天开发前看什么

1. `../checklists/每日开发检查清单.md`
2. 当前需求文档和验收标准
3. `CLAUDE.md`
4. 最近一次 CI 状态

如果今天要改权限、数据库、生产配置、密钥、AI 配置、Feature Flag 或部署流程,先找 AI 代码守护人确认。

## 提 PR 前必须确认

| 项目 | 判断标准 |
| --- | --- |
| 需求 | 没有未回答的红色/橙色问题 |
| 代码 | 没有硬编码密钥、生产 URL、临时 TODO |
| 测试 | 至少覆盖正常路径和关键异常路径 |
| 安全 | 外部输入经过 Zod 或前端校验,后端敏感操作有权限检查 |
| 文档 | 改了流程、接口、上线方式或应急方式时,同步更新文档 |
| 验证 | `npm run lint`、`npm test` 或相关命令有结论 |

## required checks 是什么

Atlas 的 main 分支受 GitHub 保护。PR 合并前必须通过:

| Check | 作用 |
| --- | --- |
| `lint` | ESLint 静态检查 |
| `security` | `npm audit --audit-level=high` |
| `test` | 前后端 Vitest、覆盖率报告、构建 |
| `e2e-core` | Playwright 核心端到端旅程 |

这些检查慢也不能跳过。最近 E2E 通常需要 20 分钟以上,这是当前体系的已知成本。

## 常见红线

- 不要把 `JWT_SECRET`、`AI_API_KEY`、数据库连接串写进代码或文档。
- 不要在没有确认的情况下创建数据库迁移或改生产配置。
- 不要因为是“小改动”就跳过 PR 和 CI。
- 不要照搬 AI 生成的路径、API、字段名。Atlas User 没有 email/phone 字段。
- 不要把 Sentry、Unleash、生产告警当成“已配置”。没有证据时记录为缺口。

## 出现 Bug 时怎么做

1. 收集现象:页面、账号角色、操作步骤、时间、浏览器、环境。
2. 收集证据:浏览器 Console、Network、server requestId、Pino 日志、Sentry issue(如有)。
3. 使用 `../prompts/05-Bug诊断提示词.md` 做根因分析。
4. 先补一个能复现问题的测试,再修代码。
5. 修复后跑相关测试,必要时跑 `npm run test:e2e:core`。
6. 如果影响生产,同步 `../qa/incident-response.md` 的事故记录要求。

## 上线前怎么判断能不能发

上线前至少确认:

- PR required checks 全绿。
- 需求验收标准已过。
- 上线前检查清单已跑。
- 如果有 Feature Flag,能关闭。
- 如果出问题,知道回滚到哪个版本、谁执行、观察哪些指标。
- 不在周五下班前、节假日前或无人值守时发布高风险变更。

## AI 代码守护人的日常动作

| 频率 | 动作 |
| --- | --- |
| 每个 PR | 看 PR 描述、CI、AI review、测试范围和风险说明 |
| 每周 | 看 E2E 时长、失败原因、依赖安全、核心模块覆盖 |
| 每月 | 执行月度质量审计,更新报告和下月重点 |
| 事故后 | 确认复盘、补测试、更新检查清单或提示词 |

## 第一次使用时的最短路线

1. 读本手册。
2. 打开 `../requirements/REQUIREMENT_TEMPLATE.md` 写一个小需求。
3. 用 `../prompts/01-需求澄清提示词.md` 审查需求。
4. 让 AI 生成改动前,把 CLAUDE.md、需求、相关代码一起提供。
5. 提 PR 前跑 `../checklists/提交代码前检查清单.md`。
