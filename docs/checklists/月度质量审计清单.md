# 月度质量审计清单

> 使用时机：每月最后一个工作日或下月第一个工作日
> 输出：月度质量报告 + 下月 Top 3-5 改进项

## 数据收集

- [ ] GitHub Actions：`main` CI 通过率、全 workflow 通过率、最近 60 次 run 趋势。
- [ ] CI 耗时：`lint`、`security`、`test`、`e2e-core` 分别耗时，重点看 E2E 是否退化。
- [ ] 测试数量：server/client 单元集成测试文件数、E2E spec 数、passed/skipped/todo 数。
- [ ] 覆盖率：`npm run test:coverage` 和 `npm run coverage:core`。
- [ ] lint：errors / warnings，总 warning 是否下降。
- [ ] 依赖：`npm audit --json`、`npm outdated --json`。
- [ ] 技术债：`TODO/FIXME/XXX`，区分模板示例和生产代码。
- [ ] PR：合并数量、失败 checks、review 等待时间、AI code review 拦截情况。

## 生产与业务数据

- [ ] 事故数：P0/P1/P2/P3。
- [ ] MTTR：从发现到恢复的平均时间。
- [ ] 缺陷逃逸率：上线后发现的 bug / 总 bug。
- [ ] 回滚次数和回滚原因。
- [ ] Sentry / 告警：Top 错误、误报、漏报。
- [ ] 客服/业务反馈：高频问题、用户影响面、业务指标异常。
- [ ] 如果以上数据源尚未接入，报告中必须标为“待业务负责人补充”，不要伪造。

## 体系健康度

按 1-5 分评估：

- [ ] 输入控制：`CLAUDE.md`、需求模板、提示词是否被实际使用。
- [ ] 自动化门禁：required checks 是否稳定、耗时是否可接受。
- [ ] 人工审查：review 是否有效，不只是形式批准。
- [ ] 生产保护：Feature Flag、灰度、回滚、应急预案是否经过演练。
- [ ] 流程文档：文档是否反映真实流程，新人是否能照着执行。
- [ ] 团队协作：业务团队是否能独立使用清单和提示词。

## 改进行动

- [ ] 用 ICE 评分选出 Top 3-5 改进项。
- [ ] 每个改进项写清 Owner、截止日期、验收标准、风险。
- [ ] 判断是否需要更新 `CLAUDE.md`、检查清单、提示词、测试策略、CI 配置。
- [ ] 发布月报到 `docs/qa/reports/monthly-quality-audit-YYYYMM.md`。
