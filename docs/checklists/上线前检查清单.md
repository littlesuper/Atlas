# 上线前检查清单

> 使用时机：生产发布前 24-48 小时启动，发布前 1 小时再过一遍
> 决策权：发布负责人 + 业务负责人 + AI 代码守护人共同 Go/No-Go

## 代码与 CI

- [ ] 发布 commit hash 已确定，并写入发布记录。
- [ ] `main` 最新 CI 全绿，required checks 为 `lint`、`security`、`test`、`e2e-core`。
- [ ] Release Notes 已写清功能、修复、风险、用户影响。
- [ ] 本次发布不包含未确认的密钥、生产配置、数据库迁移或破坏性数据操作。
- [ ] 若包含数据库 schema 变更，已有迁移计划、备份计划、回滚/向前修复策略，并经负责人确认。

## 测试与验收

- [ ] `npm run lint`、`npm run test`、`npm run test:coverage`、`npm run coverage:core` 已通过或由 CI 证明通过。
- [ ] 核心后端模块覆盖率不低于最近月报基线。
- [ ] E2E core 已通过；如改动涉及视觉/无障碍，额外跑 `test:e2e:visual` 或 `test:e2e:a11y`。
- [ ] staging 上跑过核心路径：登录、项目、活动、产品、周报、权限边界。
- [ ] 新功能有 Feature Flag 或明确说明为什么属于核心路径不能关闭。

## 配置与可观测性

- [ ] 生产环境变量由部署平台或密钥管理注入，仓库不保存真实值。
- [ ] `/api/health` 可访问，并返回当前版本。
- [ ] 日志包含 requestId，不记录密码、token、完整密钥或敏感个人信息。
- [ ] 已确认 `docs/qa/metrics.md`、`docs/qa/alerts.md`、`docs/qa/error-tracking.md` 中适用项的当前状态。
- [ ] 如果 Sentry / 告警真实项目尚未配置，发布记录中明确写出该风险和人工观察替代方案。

## 灰度与回滚

- [ ] 涉及可选模块时，按 `docs/qa/feature-flags.md` 设置默认关闭或小流量灰度。
- [ ] 灰度发布按 `docs/qa/canary-release-runbook.md` 执行，阶段为 `5% -> 25% -> 50% -> 100%`。
- [ ] 回滚方案已按 `docs/qa/rollback-runbook.md` 复核，优先关闭 Feature Flag，其次代码回滚，最后才数据库恢复。
- [ ] 发布后 24 小时值班人与升级路径已明确。
- [ ] staging 应急演练如果尚未执行，发布负责人已明确接受该风险或安排演练窗口。

## Go/No-Go

- [ ] 没有 P0/P1 未解决问题。
- [ ] 所有带病上线项都有负责人、缓解措施和回滚路径。
- [ ] 业务负责人确认发布时间和用户沟通方式。
- [ ] AI 代码守护人确认质量门禁与回滚材料齐备。
