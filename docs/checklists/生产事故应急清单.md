# 生产事故应急清单

> 使用时机：staging / production 出现异常时
> 核心原则：先止血，后追因；先记录，后修改

## 前 5 分钟

- [ ] 指定事件负责人，停止多人同时操作。
- [ ] 记录开始时间、发现渠道、影响功能、影响范围。
- [ ] 判断等级：
  - P0：系统不可用、数据损坏/泄露、安全事件。
  - P1：登录、项目、活动、产品、周报等核心路径不可用，大量用户受影响。
  - P2：非核心功能异常或局部用户受影响。
  - P3：轻微 UI / 文案 / 边缘问题。
- [ ] P0/P1 立即通知业务负责人和值班人；不要等根因完全确认。
- [ ] 打开 `/api/health`、日志、GitHub 最近发布记录、监控/告警/Sentry（如已接入）。

## 止血决策

- [ ] 如果问题与最近发布高度相关，优先按 `docs/qa/rollback-runbook.md` 准备回滚。
- [ ] 如果问题集中在可选新功能，优先关闭对应 Feature Flag。
- [ ] 如果问题来自外部服务，执行降级方案并公告受影响功能。
- [ ] 如果怀疑数据损坏，先备份现场和日志，不直接改生产数据库。
- [ ] 如果需要真实回滚、数据库恢复、生产配置修改，必须有负责人明确授权。

## 恢复验收

- [ ] `/api/health` 正常。
- [ ] 管理员可以登录。
- [ ] 项目列表和项目详情可以打开。
- [ ] 活动列表和关键编辑动作可用。
- [ ] 产品、周报、权限相关核心路径恢复。
- [ ] 错误率、延迟、用户反馈回到可接受范围。
- [ ] 业务负责人确认可以解除事故状态。

## 事后复盘

- [ ] 24-48 小时内补事故报告。
- [ ] 记录根因、触发条件、拦截失败点、恢复耗时。
- [ ] 判断是否需要更新 `CLAUDE.md`、检查清单、提示词、测试或监控。
- [ ] 如果由未演练流程导致恢复慢，安排 staging 演练。
