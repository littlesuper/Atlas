# 每日开发检查清单

> 使用频率：每个工作日、每个任务开始前
> 目标：让业务团队和 AI 在同一套边界内工作

## 上班第一件事

- [ ] 看 GitHub `main` 最新 CI 是否为绿色。
- [ ] 看当前是否有 open PR、失败 check、未处理 review。
- [ ] 看 `docs/qa/reports/monthly-quality-audit-202605.md` 中列出的未闭环风险是否影响今天工作。
- [ ] 如果已接入 Sentry / 告警平台，确认夜间是否有 P0/P1；未接入时至少检查 `/api/health` 和部署负责人反馈。
- [ ] 如有 P0/P1、main 红、生产异常，先处理异常，不开始新功能。

## 每个任务开始前

- [ ] 需求是否已写入 `docs/requirements/`，或有等价的明确说明。
- [ ] 是否使用过 `docs/prompts/01-需求澄清提示词.md` 明确验收标准。
- [ ] 是否识别了影响范围：前端、后端、Prisma、权限、测试、部署。
- [ ] 是否需要业务/产品决策；如果需要，先问清楚。
- [ ] 是否涉及密钥、生产配置、数据库迁移、真实部署或已部署服务影响；如果涉及，先暂停并确认授权。

## 开发过程中

- [ ] 先读 `CLAUDE.md`、相关代码、相关测试、`docs/qa/` 和 `docs/process/`，再让 AI 改。
- [ ] 使用 `docs/prompts/02-代码生成提示词.md`，不要让 AI 无边界发挥。
- [ ] 每次改动保持小步：一个 PR 只解决一个清楚的问题。
- [ ] 看到 AI 修改不相关文件、要求 destructive 命令、跳过测试、伪造外部服务结果时，立即停下来复核。
- [ ] 新功能默认考虑 Feature Flag；核心流程不要默认关闭。

## 下班前

- [ ] 当前分支已 push，或明确记录本地未完成状态。
- [ ] 自己的 PR 状态已看过：CI、review、冲突、待回复问题。
- [ ] 今天发现的流程痛点已记录到 `docs/process/process-change-log.md` 或下次月度审计材料。
- [ ] 如果发现重复 bug 模式，准备在 Week 8 文档完善时更新 `CLAUDE.md` / 提示词 / 清单。
